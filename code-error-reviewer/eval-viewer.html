📋 Eval 1 — Error useRouter en Next.js (Client Component)
Me aparece este error en la consola del navegador al abrir mi página de asistencia en Next.js: Error: useRouter only works in Client Components. Add the 'use client' directive at the top of the file to use it. at AsistenciaPage (./src/app/asistencia/page.tsx:23:45) Mi código es: ``` import { useRouter } from 'next/navigation'; import { useState } from 'react'; export default function AsistenciaPage() { const router = useRouter(); const [mes, setMes] = useState(0); return <div>Asistencia</div>; } ```
CON Skill
🔍 Diagnóstico El componente AsistenciaPage usa hooks de React (useRouter, useState), pero en Next.js 14 con App Router todos los componentes son Server Components por defecto. Los hooks solo funcionan en Client Components, y el error te lo dice directamente: falta la directiva 'use client' al inicio del archivo. 💡 Solución 
'use client';  // ← AGREGAR ESTA LÍNEA

import { useRouter } from 'next/navigation';
import { useState } from 'react';

export default function AsistenciaPage() {
  const router = useRouter();
  const [mes, setMes] = useState(0);
  return <div>Asistencia</div>;
}
Solo agrega 'use client'; como primera línea del archivo (antes de cualquier import). Eso es todo. 📌 Por qué pasó En el App Router de Next.js 13+, el modelo por defecto es que los componentes se renderizan en el servidor. Los hooks como useRouter, useState, useEffect, etc. dependen del ciclo de vida del cliente, así que Next.js los bloquea en Server Components. ⚡ Tips extra - La directiva 'use client' debe ir en la primera línea del archivo, antes de cualquier import. - Si tu página tiene partes que no necesitan interactividad, considera separar el componente interactivo en un archivo aparte. Mejora el rendimiento. - Otros hooks que también requieren 'use client': usePathname, useSearchParams, useParams, useContext.
SIN Skill (baseline)
CAUSA DEL PROBLEMA: En Next.js 13+ con el App Router, todos los componentes son Server Components por defecto. Los hooks como useRouter y useState son exclusivos de Client Components. SOLUCIÓN: Agrega la directiva 'use client' como la primera línea del archivo: 
'use client';

import { useRouter } from 'next/navigation';
import { useState } from 'react';

export default function AsistenciaPage() {
  const router = useRouter();
  const [mes, setMes] = useState(0);
  return <div>Asistencia</div>;
}
REGLA GENERAL: Siempre que uses hooks de React, hooks de Next.js del lado del cliente, manejadores de eventos o APIs del navegador, debes agregar 'use client' al inicio del archivo. NOTAS ADICIONALES: - La directiva debe ser la primera línea — antes de los imports. - Cuando un componente tiene 'use client', todos sus hijos también se convierten en Client Components. - Considera extraer solo la parte interactiva en un componente separado con 'use client' para mejor rendimiento.
📋 Eval 2 — Supabase devuelve array vacío (case-sensitive)
Estoy haciendo una consulta a Supabase y siempre me retorna [] aunque sé que hay datos. Mi código: const { data, error } = await supabase .from('deportistas') .select('*') .eq('proyecto', 'SUB 8A'); console.log(data); // [] siempre vacío console.log(error); // null En Supabase Studio veo los registros y el campo proyecto dice 'Sub 8a'.
CON Skill
🔍 Diagnóstico El filtro .eq('proyecto', 'SUB 8A') no encuentra registros porque PostgreSQL hace comparaciones de texto de forma case-sensitive por defecto. El valor en la BD es 'Sub 8a' y buscas 'SUB 8A' — son cadenas distintas para Postgres. 💡 Solución Opción 1 — La más simple: usar el valor exacto tal como está en la BD. 
// Antes:
.eq('proyecto', 'SUB 8A')
// Después:
.eq('proyecto', 'Sub 8a') // valor exacto como aparece en Supabase Studio
Opción 2 — Si los datos son inconsistentes en la BD, usa ilike para búsqueda case-insensitive: 
.ilike('proyecto', 'sub 8a') // no importa mayúsculas/minúsculas
📌 Por qué pasó PostgreSQL trata los textos con el operador = de forma estrictamente case-sensitive: 'SUB 8A' = 'Sub 8a' evalúa como false. Supabase traduce .eq() directamente a = en SQL. El error no produce ningún mensaje porque la consulta es válida — simplemente no encuentra filas y devuelve [] con error: null. ⚡ Tips extra 1. Normaliza los datos en origen — guarda siempre en un formato canónico. 2. Si usas ilike frecuentemente, crea un índice funcional: CREATE INDEX ON deportistas (LOWER(proyecto)). 3. Consejo debug: cuando Supabase devuelve [] con error: null, el primer sospechoso es siempre el filtro. Prueba primero sin .eq() para confirmar que los datos cargan.
SIN Skill (baseline)
El problema es de sensibilidad a mayúsculas y minúsculas (case-sensitivity). CAUSA: PostgreSQL (y por lo tanto Supabase) es case-sensitive en sus comparaciones de texto por defecto. Tu código busca 'SUB 8A' pero en la base de datos está guardado como 'Sub 8a'. Para PostgreSQL son valores distintos. SOLUCIONES: Opción 1 — Usar el valor exacto: 
.eq('proyecto', 'Sub 8a')  // valor exacto del campo en BD
Opción 2 — Usar ilike para case-insensitive: 
.ilike('proyecto', 'sub 8a')
Opción 3 — Normalizar al insertar, guardando siempre en mayúsculas o minúsculas. NOTA IMPORTANTE: Que error sea null con data: [] es el comportamiento correcto cuando la consulta es válida pero no encuentra registros. No es un error de Supabase — es un caso de no-match. También verifica que no tengas RLS (Row Level Security) activado en la tabla deportistas, ya que podría estar bloqueando las consultas incluso con el valor correcto.
📋 Eval 3 — TypeScript "Property is missing" en interfaz
TypeScript me da este error: Type '{ id: string; nombre: string; proyecto: string | null; }' is not assignable to type 'Deportista'. Property 'sede' is missing in type '{ ... }' but required in type 'Deportista'. Mi interfaz es: interface Deportista { id: string; nombre: string; proyecto: string | null; sede: string; fechaNacimiento?: string; } const dep: Deportista = { id: '123', nombre: 'Carlos', proyecto: 'SUB 8A', // falta sede };
CON Skill
🔍 Diagnóstico El objeto asignado a dep no incluye la propiedad sede, pero en la interfaz Deportista esa propiedad está declarada como requerida (sin ?). TypeScript exige que todos los campos obligatorios estén presentes en el objeto. 💡 Solución Tienes dos caminos según tu intención: Opción A — Agregar sede al objeto: 
const dep: Deportista = {
  id: '123',
  nombre: 'Carlos',
  proyecto: 'SUB 8A',
  sede: 'Medellín',  // ← agregar
};
Opción B — Hacer sede opcional en la interfaz (solo si un deportista puede no tener sede): 
interface Deportista {
  // ...
  sede?: string;   // ← añadir el ?
  // ...
}
📌 Por qué pasó TypeScript usa tipado estructural: para que un objeto sea válido como Deportista, debe tener TODAS las propiedades requeridas. sede no tiene ?, así que es obligatoria. fechaNacimiento sí tiene ? por eso no da error aunque falte. ⚡ Tips extra - Partial<Deportista> — si necesitas construir el objeto de forma incremental: const depParcial: Partial<Deportista> = { id: '123', nombre: 'Carlos' }; - El mensaje de error de TypeScript es tu amigo: "Property 'sede' is missing in type... but required in type 'Deportista'" te dice exactamente qué falta y en qué tipo.
SIN Skill (baseline)
El error de TypeScript indica que al objeto que estás asignando a dep le falta la propiedad sede, que es requerida en la interfaz Deportista. EXPLICACIÓN: - sede: string → sin ?, es REQUERIDA — debe estar siempre presente. - sede?: string → con ?, es OPCIONAL — puede omitirse. - fechaNacimiento?: string → opcional, por eso no da error aunque no esté en el objeto. SOLUCIÓN 1 — Agregar la propiedad al objeto: 
const dep: Deportista = {
  id: '123',
  nombre: 'Carlos',
  proyecto: 'SUB 8A',
  sede: 'Sede Norte',  // añadir esto
};
SOLUCIÓN 2 — Hacer sede opcional en la interfaz (si tiene sentido que un deportista pueda no tener sede): 
interface Deportista {
  id: string;
  nombre: string;
  proyecto: string | null;
  sede?: string;        // agregar "?"
  fechaNacimiento?: string;
}
Cuál opción usar depende de tu lógica de negocio: si todo deportista DEBE tener una sede, usa la Solución 1. Si puede registrarse sin sede inicial, usa la Solución 2. También puedes usar Partial<Deportista> si construyes el objeto en pasos: 
const depParcial: Partial<Deportista> = { id: '123' };
const dep: Deportista = { ...depParcial, sede: 'Bello', proyecto: 'SUB 8A', nombre: 'Carlos' };
